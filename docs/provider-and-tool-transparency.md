# Provider and Tool Transparency

## This is not a bypass pack

This template is intended to make legitimate authorization clearer. It is not a bypass pack, jailbreak pack, or moderation-evasion pack.

It does not guarantee that an AI model, security tool, cloud provider, hosting provider, browser, extension store, code host, or platform will allow a request.

A provider may still flag, narrow, refuse, rate-limit, review, or block security-related activity.

## Why transparency helps

Clear authorization context can help a legitimate tool understand:

- who owns the work;
- what assets are in scope;
- what test classes are allowed;
- what is excluded;
- what data must not be touched;
- when to stop.

That reduces ambiguity. It does not remove provider safety obligations.

## Correct behavior when blocked

When blocked or refused:

- respect the refusal;
- do not rephrase to hide intent;
- do not claim false authorization;
- do not omit relevant target or safety context;
- narrow to passive defensive work;
- ask the accountable owner for scope clarification if needed.

## Safer alternatives

Use these instead of active or risky actions:

- review source code;
- review configuration;
- review dependency manifests;
- write a threat model;
- draft a finding report;
- draft a remediation plan;
- design a synthetic unit test;
- design a staging-only validation plan;
- redact evidence;
- update the rules of engagement.

## Prompt transparency principle

A good security prompt should be honest about:

- authorization;
- scope;
- exclusions;
- environment;
- data-handling limits;
- whether testing is passive or active.

Do not use this template to disguise intent.
